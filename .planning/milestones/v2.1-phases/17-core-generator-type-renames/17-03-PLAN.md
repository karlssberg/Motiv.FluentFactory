## Rename Mapping (NAME-04, locked decision)

The two existing types and their concrete responsibilities (extracted from the source by the planner before this plan was written):

| Old name | New name | Why |
|----------|----------|-----|
| `FluentFactoryMethodDeclaration` | **`StepTerminalMethodDeclaration`** | Called from `FluentStepDeclaration.cs` for the `CreationMethod` switch arm. Builds the terminal/creation method that lives **inside a step struct**. Reads parameter values from `this.{field}` accessors (already-stored fields on the step struct). Threads tuple-field repacking through `RepackTupleFieldArgument`. The "Step" prefix mirrors its sibling `FluentStepMethodDeclaration` and tells the reader: "this declaration lives on a step." The "Terminal" qualifier distinguishes it from `FluentStepMethodDeclaration` (which emits *transition* methods between steps). |
| `FluentRootFactoryMethodDeclaration` | **`RootTerminalMethodDeclaration`** | Called from `RootTypeDeclaration.BuildRootMethodSyntax` for the `{ Return: TargetTypeReturn }` switch arm. Builds the terminal/creation method that lives **directly on the root type**, used when the user can construct the target in a single call from the root (no step intermediates needed). Takes parameters via parameter list rather than reading fields, because the root has no field storage yet. The "Root" prefix tells the reader: "this declaration lives on the root." Symmetric with `StepTerminalMethodDeclaration`. |

The shared word "Terminal" aligns with the existing `TerminalMethod` / `TerminalMethodKind` enum (the public attribute property that controls terminal-method generation), so the vocabulary is internally consistent: `[FluentRoot(TerminalMethod = TerminalMethod.FixedName)]` configures behavior that is implemented by `StepTerminalMethodDeclaration` and `RootTerminalMethodDeclaration`.

This rename mapping is a **locked decision**. Executors do not revisit it.

## Phase 18 Handoff: Grep Pattern Refinement

The Phase 17 zero-hit gate (success criterion #5) uses a **word-boundary regex refinement** of the plain alternation pattern in ROADMAP.md:

- **ROADMAP.md (plain alternation):** `git grep -n "FluentFactoryGenerator\|FluentFactoryCompilationUnit\|FluentFactoryMetadata\|FluentFactoryDefaults\|FluentFactoryMethodDeclaration" -- src/Converj.Generator/`
- **Phase 17 gate (word-boundary regex):** `git grep -nP "(?<![A-Za-z])(FluentFactoryGenerator|FluentFactoryCompilationUnit|FluentFactoryMetadata|FluentFactoryDefaults|FluentFactoryMethodDeclaration)(?![A-Za-z])" -- src/Converj.Generator/`

**Why the refinement is necessary:** Phase 17 deliberately defers three method-name renames to Phase 18, because those method identifiers are tangled with the `*Factory` → `*Builder` renames that are Phase 18's actual scope:

- `CreateFluentFactoryCompilationUnit` — on `FluentModelFactory`, which itself is renamed in Phase 18
- `GetFluentFactoryMetadata` — on the renamed `FluentRootMetadataReader`
- `GetFluentFactoryDefaults` — on the renamed `FluentRootMetadataReader`

The plain alternation from ROADMAP is a plain-text substring match, so it would substring-hit those deferred method identifiers (e.g., `CreateFluentFactoryCompilationUnit` contains the substring `FluentFactoryCompilationUnit`). Running the plain alternation at the end of Phase 17 would produce non-zero hits at minimum in `FluentModelFactory.cs`, `FluentRootGenerator.cs`, `FluentRootMetadataReader.cs`, `FluentTargetValidator.cs`, and `FluentTargetContextFactory.cs`. None of those would be type-name regressions — they would be the intentionally-deferred method identifiers.

The negative look-around `(?<![A-Za-z])...(?![A-Za-z])` enforces that only standalone type-name tokens are matched, not substrings inside `Create*`/`Get*` method identifiers. This is semantically equivalent to the ROADMAP intent (zero hits for the legacy type names) while honoring the Phase 17 / Phase 18 boundary.

**Phase 18 obligation:** Once Phase 18 completes the three method-name renames (as part of or alongside the `*Factory` → `*Builder` work), Phase 18 MUST run the **plain alternation** from ROADMAP.md as its own final gate, with no word-boundary refinement. That is the moment at which the ROADMAP success criterion #5 can be satisfied verbatim, and the refinement can be retired.
