# Phase 6: Generated Code Hardening - Context

**Gathered:** 2026-03-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Harden generated `.g.cs` output for robust consumption in any namespace context. Two changes: (1) fully qualify all type references with `global::` prefix, and (2) add `[GeneratedCode]` attribution to all generated type declarations. No new API features or runtime changes.

</domain>

<decisions>
## Implementation Decisions

### global:: Qualification Scope
- Every concrete type reference in generated output uses `global::` prefix — parameters, return types, fields, base types, attributes, constraint types
- C# built-in type aliases (int, string, bool, etc.) stay as language keywords — they can't conflict
- Generic type parameters (T, TResult, etc.) stay unqualified — they're locally declared; only their constraint types get `global::`
- Replace the existing `ToDynamicDisplayString` namespace-stripping logic entirely with always-global qualification — remove the dynamic namespace comparison path

### GeneratedCode Attribute Placement
- All generated type declarations get `[GeneratedCode]` — both the partial root type extension and step structs
- Top-level types only — methods, properties, and constructors inside generated types do not get the attribute (standard .NET convention)
- Version string sourced from the generator assembly version at build time — auto-syncs with NuGet package version
- The attribute reference itself uses `global::` prefix: `[global::System.CodeDom.Compiler.GeneratedCode("Motiv.FluentFactory", "version")]`

### Using Directives Strategy
- Remove all using directives from generated output — with `global::` everywhere, usings are unnecessary
- Emit `#nullable enable` at the top of generated files
- Emit `// <auto-generated/>` header comment at the top of generated files

### Test Update Approach
- Bulk update all test expected outputs in a single pass after generation changes are complete
- Add a dedicated namespace conflict test using the existing `CSharpSourceGeneratorVerifier` — source with conflicting type names (e.g., a user type named `String` or `List`) proves `global::` prevents compilation errors
- Version string tests verify non-empty and well-formed (semver-like pattern), not pinned to exact version

### Claude's Discretion
- Exact refactoring approach for replacing `ToDynamicDisplayString` with global-qualified output
- How to read assembly version at generator compile time
- Ordering of changes within the implementation

</decisions>

<specifics>
## Specific Ideas

- Success criterion #1 explicitly requires "a consumer project with conflicting type names" — the conflict test directly validates this
- The `CompilationUnit.cs` currently builds using directives from `file.Usings` — this entire section can be removed
- `SymbolExtensions.ToDynamicDisplayString` is the core method that strips namespace prefixes — this is the primary refactoring target

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `CSharpSourceGeneratorVerifier`: Existing test infrastructure for verifying generator output — use for the new conflict test
- `CompilationUnit.CreateCompilationUnit`: Central method assembling the generated syntax tree — entry point for removing usings and adding header/nullable directives

### Established Patterns
- Generation uses Roslyn `SyntaxFactory` to build syntax trees, then calls `NormalizeWhitespace().ToString()` for final output
- Tests use inline expected output strings compared against generator output
- Type display is handled through `SymbolExtensions.ToDynamicDisplayString` with namespace-aware stripping

### Integration Points
- `FluentFactoryGenerator.Execute` calls `CompilationUnit.CreateCompilationUnit` then `context.AddSource` — the output pipeline
- `SymbolExtensions.ToDynamicDisplayString` is called throughout Generation/ to render type names — all call sites affected by global:: change
- `RootTypeDeclaration`, `FluentStepDeclaration`, `ExistingPartialTypeStepDeclaration` — type declarations where `[GeneratedCode]` attribute needs to be added

</code_context>
