# Phase 6: Generated Code Hardening - Research

**Researched:** 2026-03-09
**Domain:** Roslyn source generator output hardening (global:: qualification + GeneratedCode attribution)
**Confidence:** HIGH

## Summary

This phase hardens the generated `.g.cs` output from the Motiv.FluentFactory source generator by (1) fully qualifying all type references with `global::` prefix and (2) decorating all generated type declarations with `[GeneratedCode]` attribute. The codebase already uses Roslyn's `SymbolDisplayFormat.FullyQualifiedFormat` in several constraint-generation spots, so the key Roslyn API is proven within this project. The primary refactoring target is `SymbolExtensions.ToDynamicDisplayString`, which currently does namespace-relative stripping -- this must be replaced with `global::` qualification. Additionally, all using directives must be removed from generated output since `global::` makes them unnecessary.

The generator targets `netstandard2.0` and uses Roslyn `SyntaxFactory` to build syntax trees. There are ~20 call sites for `ToDynamicDisplayString` spread across Generation and Model layers. The `CompilationUnit.CreateCompilationUnit` method currently builds using directives from `file.Usings` -- this entire section gets removed. Three type declaration classes (`RootTypeDeclaration`, `FluentStepDeclaration`, `ExistingPartialTypeStepDeclaration`) need `[GeneratedCode]` attribute addition. Approximately 15 test files contain inline expected output strings that will need bulk update.

**Primary recommendation:** Replace `ToDynamicDisplayString` with a new method that uses `SymbolDisplayFormat.FullyQualifiedFormat` (which emits `global::` prefix natively), remove using directive generation from `CompilationUnit`, add `[GeneratedCode]` attributes to the three type declaration factories, and emit `// <auto-generated/>` + `#nullable enable` headers.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- Every concrete type reference in generated output uses `global::` prefix -- parameters, return types, fields, base types, attributes, constraint types
- C# built-in type aliases (int, string, bool, etc.) stay as language keywords -- they can't conflict

## Architecture Patterns

### Current Type Rendering Pipeline
```
ITypeSymbol
  -> ToDynamicDisplayString(currentNamespace)  [strips namespace if same]
  -> IdentifierName(string)                    [creates Roslyn syntax node]
  -> emitted in generated .g.cs output
```

### Target Type Rendering Pipeline
```
ITypeSymbol
  -> ToGloballyQualifiedDisplayString()  [always emits global:: prefix]
  -> ParseTypeName(string) or IdentifierName(string)  [creates Roslyn syntax node]
  -> emitted in generated .g.cs output (no using directives needed)
```

### Pattern 1: Using SymbolDisplayFormat.FullyQualifiedFormat
**What:** Roslyn's built-in format that emits `global::` prefix for all type references
**When to use:** Every place a type symbol needs to be rendered as a string for code generation
**Example:**
```csharp
// Roslyn built-in - already used in constraint generation within this project
// SymbolDisplayFormat.FullyQualifiedFormat includes global:: by default
var display = typeSymbol.ToDisplayString(SymbolDisplayFormat.FullyQualifiedFormat);
// Result: "global::System.Collections.Generic.List<global::System.String>"

// To get special types as keywords (int instead of global::System.Int32):
var format = SymbolDisplayFormat.FullyQualifiedFormat
    .WithMiscellaneousOptions(
        SymbolDisplayMiscellaneousOptions.UseSpecialTypes |
        SymbolDisplayMiscellaneousOptions.IncludeNullableReferenceTypeModifier);
var display = typeSymbol.ToDisplayString(format);
// Result for List<int>: "global::System.Collections.Generic.List<int>"
// Result for string: "string" (kept as keyword)
```

### Pattern 2: GeneratedCode Attribute via SyntaxFactory
**What:** Adding `[GeneratedCode]` attribute to type declarations using Roslyn SyntaxFactory
**Example:**
```csharp
// Build the attribute syntax
var generatedCodeAttribute = SyntaxFactory.Attribute(
    SyntaxFactory.ParseName("global::System.CodeDom.Compiler.GeneratedCode"),
    SyntaxFactory.AttributeArgumentList(
        SyntaxFactory.SeparatedList(new[]
        {
            SyntaxFactory.AttributeArgument(
                SyntaxFactory.LiteralExpression(
                    SyntaxKind.StringLiteralExpression,
                    SyntaxFactory.Literal("Motiv.FluentFactory"))),
            SyntaxFactory.AttributeArgument(
                SyntaxFactory.LiteralExpression(
                    SyntaxKind.StringLiteralExpression,
                    SyntaxFactory.Literal(version)))
        })));

// Add to type declaration
typeDeclaration = typeDeclaration.WithAttributeLists(
    typeDeclaration.AttributeLists.Add(
        SyntaxFactory.AttributeList(
            SyntaxFactory.SingletonSeparatedList(generatedCodeAttribute))));
```

### Pattern 3: Auto-generated Header and Nullable Enable
**What:** Adding `// <auto-generated/>` comment and `#nullable enable` directive at the top of generated files
**Example:**
```csharp
// In CompilationUnit.CreateCompilationUnit, add leading trivia:
var autoGeneratedComment = SyntaxFactory.Comment("// <auto-generated/>");
var nullableEnable = SyntaxFactory.Trivia(
    SyntaxFactory.NullableDirectiveTrivia(
        SyntaxFactory.Token(SyntaxKind.EnableKeyword), true));

compilationUnit = compilationUnit.WithLeadingTrivia(
    autoGeneratedComment,
    SyntaxFactory.CarriageReturnLineFeed,
    nullableEnable,
    SyntaxFactory.CarriageReturnLineFeed);
```

### Pattern 4: Reading Assembly Version at Compile Time
**What:** Getting the generator assembly version for the `GeneratedCode` attribute
**Example:**
```csharp
// Option A: Read from assembly at runtime (generator runs at compile time of consumer)
var version = typeof(FluentFactoryGenerator).Assembly.GetName().Version?.ToString() ?? "0.0.0";

// Option B: Use AssemblyInformationalVersion for semver-like string
var version = typeof(FluentFactoryGenerator).Assembly
    .GetCustomAttribute<System.Reflection.AssemblyInformationalVersionAttribute>()
    ?.InformationalVersion ?? "0.0.0";
```

### Anti-Patterns to Avoid
- **Partial global:: qualification:** Do not mix `global::` and namespace-relative references in the same generated file. Once using `global::`, use it everywhere for consistency.
- **Keeping using directives:** With `global::` everywhere, using directives are dead code. Remove them entirely.
- **Qualifying type parameters:** Generic type parameters like `T`, `TResult` are locally declared and must NOT get `global::` prefix. Only their constraint types (e.g., `where T : global::System.IComparable`) get qualified.
- **Qualifying C# keywords:** Built-in aliases (`int`, `string`, `bool`, `object`, `decimal`, etc.) should remain as keywords. Use `SymbolDisplayMiscellaneousOptions.UseSpecialTypes` to preserve this.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Global type qualification | Custom string manipulation to prepend `global::` | `SymbolDisplayFormat.FullyQualifiedFormat` | Handles generics, nested types, arrays, nullable types correctly. Already proven in this codebase for constraint generation. |
| Detecting C# keywords | Manual list of keyword type names | `SymbolDisplayMiscellaneousOptions.UseSpecialTypes` | Built into Roslyn's display format, handles all 15+ special type aliases |
| Auto-generated header | Manual string concatenation | `SyntaxFactory.Comment()` + trivia APIs | Proper trivia handling ensures correct whitespace in output |

## Common Pitfalls

### Pitfall 1: IdentifierName vs ParseTypeName for global:: Strings
**What goes wrong:** `SyntaxFactory.IdentifierName("global::System.String")` treats the entire string as a single identifier token, producing invalid syntax.
**Why it happens:** `IdentifierName` creates a simple name node, not a qualified name. It does not parse dots or `global::`.
**How to avoid:** Use `SyntaxFactory.ParseTypeName("global::System.String")` which correctly parses the string into a `QualifiedNameSyntax` with `AliasQualifiedNameSyntax` for the `global::` part.
**Warning signs:** Generated code compiles but has syntax like `global::System.String` as a single token rather than a qualified name.

### Pitfall 2: Special Types in FullyQualifiedFormat
**What goes wrong:** Without `UseSpecialTypes`, `int` renders as `global::System.Int32`, `string` as `global::System.String`.
**Why it happens:** `SymbolDisplayFormat.FullyQualifiedFormat` does NOT include `UseSpecialTypes` by default.
**How to avoid:** Create a custom format that adds `UseSpecialTypes`:
```csharp
private static readonly SymbolDisplayFormat GlobalFormat = SymbolDisplayFormat.FullyQualifiedFormat
    .WithMiscellaneousOptions(
        SymbolDisplayMiscellaneousOptions.UseSpecialTypes |
        SymbolDisplayMiscellaneousOptions.IncludeNullableReferenceTypeModifier);
```
**Warning signs:** Test output shows `global::System.Int32` instead of `int`.

### Pitfall 3: Constraint Types Already Use FullyQualifiedFormat (Without global::)
**What goes wrong:** The existing constraint generation code in `RootTypeDeclaration`, `FluentStepDeclaration`, and `FluentStepMethodDeclaration` already uses `SymbolDisplayFormat.FullyQualifiedFormat` but explicitly strips `global::` with `.WithGlobalNamespaceStyle(SymbolDisplayGlobalNamespaceStyle.Omitted)`. This must be changed to include `global::`.
**Why it happens:** The old code relied on using directives to resolve namespaces, so `global::` was unnecessary.
**How to avoid:** Remove the `.WithGlobalNamespaceStyle(SymbolDisplayGlobalNamespaceStyle.Omitted)` override from constraint type rendering. The default `FullyQualifiedFormat` already includes `global::`.
**Warning signs:** Constraint types like `where T : System.Numerics.INumber<T>` appear without `global::` prefix.

### Pitfall 4: Step Struct Names Should NOT Get global:: Prefix
**What goes wrong:** Step struct names like `Step_0__Test_Factory` are generated identifiers in the same namespace -- they should not be `global::` qualified when referenced within the same file.
**Why it happens:** Overzealous application of `global::` to all type references, including internally generated step types.
**How to avoid:** Step struct names are internally generated and always in the same namespace as the generated code. The `IdentifierDisplayString` method for `RegularFluentStep` builds the name from `GetStepName` which creates the `Step_N__` format. These are local references within the same generated file and do not need `global::`. Only external type references (types from consumer code, BCL types, constraint types) need qualification.
**Warning signs:** Generated code has `new global::Step_0__Test_Factory(...)` which would fail to compile since step structs are in the consumer's namespace.

### Pitfall 5: Bulk Test Update Scope
**What goes wrong:** Missing test updates cause cascading test failures that obscure real problems.
**Why it happens:** ~15 test files contain inline expected output strings. Every test that checks generated output will need its expected string updated.
**How to avoid:** Make all generation changes first, then update all tests in a single systematic pass. Run the full test suite to catch any missed updates.
**Warning signs:** Dozens of test failures after the generation change, all showing the same pattern of missing `global::` or unexpected `[GeneratedCode]`.

## Code Examples

### Replacing ToDynamicDisplayString

Current usage pattern (appears ~20 times across codebase):
```csharp
// Current: namespace-relative display
parameter.ParameterSymbol.Type.ToDynamicDisplayString(method.RootNamespace)

// New: globally qualified display
// Replace with a new method or direct call:
parameter.ParameterSymbol.Type.ToGlobalDisplayString()
```

Recommended new method in `SymbolExtensions`:
```csharp
private static readonly SymbolDisplayFormat GlobalQualifiedFormat = SymbolDisplayFormat.FullyQualifiedFormat
    .WithMiscellaneousOptions(
        SymbolDisplayMiscellaneousOptions.UseSpecialTypes |
        SymbolDisplayMiscellaneousOptions.IncludeNullableReferenceTypeModifier);

/// <summary>
/// Returns the globally qualified display string for a type symbol.
/// Uses global:: prefix for all concrete types. C# keyword aliases (int, string, etc.)
/// are preserved as keywords. Type parameters (T, TResult) are returned unqualified.
/// </summary>
public static string ToGlobalDisplayString(this ITypeSymbol typeSymbol)
{
    return typeSymbol switch
    {
        ITypeParameterSymbol tp => tp.Name,  // T stays as T
        IArrayTypeSymbol arrayType => $"{arrayType.ElementType.ToGlobalDisplayString()}[]",
        _ => typeSymbol.ToDisplayString(GlobalQualifiedFormat)
    };
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Namespace-relative type names + using directives | `global::` prefix everywhere, no usings | Standard .NET practice for source generators since .NET 6+ | Eliminates all namespace collision bugs |
| No attribution on generated code | `[GeneratedCode]` attribute on all generated types | Standard .NET convention | Tools (analyzers, code coverage) correctly identify generated code |
| No auto-generated header | `// <auto-generated/>` header | Standard .NET convention | IDEs suppress warnings/suggestions in generated files |

**Context:** The `global::` prefix pattern is the established best practice for .NET source generators. The Roslyn team's own generators (e.g., `System.Text.Json`, `System.Text.RegularExpressions`) all use this pattern. The `SymbolDisplayFormat.FullyQualifiedFormat` API was specifically designed for this use case.

## Open Questions

1. **AggressiveInlining Attribute Qualification**
   - What we know: The existing `AggressiveInliningAttributeSyntax.Create()` generates `[System.Runtime.CompilerServices.MethodImpl(...)]`. With the removal of using directives, this needs `global::` prefix too.
   - What's unclear: Whether `AggressiveInliningAttributeSyntax.Create()` already uses a fully qualified string or relies on usings.
   - Recommendation: Check and update `AggressiveInliningAttributeSyntax.cs` to use `global::` prefix.

2. **Step Struct Cross-Namespace References**
   - What we know: When steps span multiple namespaces (as in `NamespaceTests`), step structs are referenced across namespace boundaries.
   - What's unclear: Whether step struct references should use `global::` when referenced from a different namespace within the same generated file.
   - Recommendation: Step structs referenced from a different namespace should use `global::` prefix since there are no using directives. Within the same namespace, the unqualified name suffices.

3. **IdentifierDisplayString Method Signature Change**
   - What we know: `IdentifierDisplayString(INamespaceSymbol currentNamespace)` takes a namespace parameter for relative resolution. With `global::` this parameter may become unnecessary.
   - What's unclear: Whether to change the signature or just change the implementation to ignore the namespace parameter.
   - Recommendation: Simplify the signature by removing the namespace parameter where possible, or change the implementation to always return globally qualified names. The `IFluentReturn` interface will need updating.
